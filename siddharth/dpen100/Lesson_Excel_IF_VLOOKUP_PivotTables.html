0 Why these three tools?

Everything in the Fundamentals lesson, formulas, functions, formatting, charts, works on data you already know the shape of. These three tools solve a different problem: making a decision automatically (IF), cross-referencing two related tables (VLOOKUP/XLOOKUP), and summarising a large flat dataset into a report (PivotTables), the three things every real production, QAQC, or cost workbook actually needs, and the three things a plain SUM/AVERAGE formula genuinely cannot do on its own.

They're also, not coincidentally, the three ideas that map most directly onto what comes after Excel: IF becomes a conditional in Python; VLOOKUP becomes a table join or a .map() in pandas; a PivotTable becomes a .groupby(). Getting comfortable with the thinking here, not the exact syntax, pays off twice.

1 The IF function

IF is Excel's basic decision-maker: test a condition, return one value if it's true, a different value if it's false. This single idea underpins an enormous amount of real mining engineering spreadsheet work, most obviously, deciding whether a sample is ore or waste.

Key idea

$$\texttt{=IF(logical\_test, value\_if\_true, value\_if\_false)}$$

The logical test almost always uses a comparison operator: =, >, <, >=, <=, <> (not equal to).

Worked example

A gold grade (g/t) is in B2. The economic cutoff grade is 0.5 g/t. Write a formula in C2 that returns "Ore" if the grade meets or exceeds cutoff, "Waste" otherwise, and evaluate it for B2 = 0.72.

$$\texttt{C2: =IF(B2>=0.5,"Ore","Waste")}$$

$0.72\ge0.5$, the condition is true.

"Ore". This single formula, dragged down a column of thousands of assay results, is the literal mechanism behind separating ore from waste in a real block model, the entire economics of a mine plan sits on top of a formula this simple, repeated at scale.

Practice 1

Using the same cutoff grade of 0.5 g/t, evaluate =IF(B2>=0.5,"Ore","Waste") for B2 = 0.49 and for B2 = 0.5 exactly.

Solution

B2 = 0.49: "Waste" (just below cutoff). B2 = 0.5: "Ore", because >= includes equality, a sample sitting exactly on the cutoff is classified as ore, not waste.

Practice 2

An equipment log has "Hours Since Last Service" in B5. Write an IF formula that returns "Overdue" if this exceeds 500 hours, "OK" otherwise.

Solution

=IF(B5>500,"Overdue","OK")

Practice 3

Evaluate the Practice 2 formula for B5 = 620 and for B5 = 500 exactly.

Solution

B5 = 620: "Overdue". B5 = 500: "OK", since the condition uses strict >, not >=, so exactly 500 hours does not yet count as overdue. (Compare with Practice 1, where >= included the boundary, worth noticing the difference is a deliberate choice each time you write a condition, not a default.)

Practice 4

QAQC duplicate check: an original assay result is in B2 (2.45 g/t), its field duplicate in C2 (2.78 g/t). Write a formula in D2 for the percentage difference, $\left|\dfrac{\text{original}-\text{duplicate}}{\text{original}}\right|\times100$, then a formula in E2 that returns "FAIL" if D2 exceeds 10%, "PASS" otherwise. Evaluate both.

Solution

D2: =ABS((B2-C2)/B2)*100

$$\left|\frac{2.45-2.78}{2.45}\right|\times100 \approx 13.47\%$$

E2: =IF(D2>10,"FAIL","PASS") → since $13.47>10$: "FAIL", this pair of samples has failed the standard duplicate-precision check and would typically trigger a re-assay or a batch review.

Practice 5

Geotechnical: a factor of safety (FoS) calculation for a pit slope is in B3. Write an IF formula that returns "Investigate" if FoS is below 1.5 (the typical minimum design threshold), "Stable" otherwise. Evaluate for B3 = 1.2 and B3 = 1.8.

Solution

=IF(B3<1.5,"Investigate","Stable")

B3 = 1.2: "Investigate" (below threshold, a genuine geotechnical flag). B3 = 1.8: "Stable".

Practice 6

Explain, with a concrete example, why mixing up > and >= in an IF condition is a genuinely dangerous mistake in a production spreadsheet, not just a minor stylistic slip.

Solution

A sample sitting exactly on a boundary value is not a rare edge case in real data, cutoff grades, service thresholds, and QAQC limits are often round numbers precisely because they're policy decisions, and real samples land on them regularly. Practice 1 and Practice 3 showed the two operators give opposite classifications for a value sitting exactly at the boundary, one calls it Ore, the other calls it Waste, purely from a single missing equals sign in the formula. In a real block model with thousands of samples, an accidental > where >= was intended could misclassify a meaningful number of boundary-grade blocks, directly affecting the reported ore reserve, exactly the kind of silent, hard-to-spot error this lesson keeps returning to.

Practice 7

Write an IF formula that checks if a blast powder factor (kg/BCM) in B4 is outside the acceptable design range of 0.3 to 0.5, returning "Out of spec" if so, "OK" if within range. (Hint: you'll need OR nested inside the IF, or two separate comparisons combined.)

Solution

=IF(OR(B4<0.3,B4>0.5),"Out of spec","OK")

The OR() function returns TRUE if either sub-condition is true (too low, or too high), and that combined TRUE/FALSE result becomes the single logical test the IF function needs. This is a natural extension worth knowing exists, even though OR and AND aren't covered in full detail in this lesson.

2 Nested IF and IFS

Real classification almost never has just two outcomes. Ore isn't just "ore or waste," it's often split into grade tiers (High, Medium, Low, Waste) that feed different processing decisions. A single IF only ever gives two outcomes, so multiple thresholds need either nested IF statements (an IF inside another IF's false branch) or the newer, flatter IFS function.

Key idea

Nested IF: $$\texttt{=IF(cond1,result1,IF(cond2,result2,IF(cond3,result3,result4)))}$$

IFS (cleaner for 3+ outcomes): $$\texttt{=IFS(cond1,result1,cond2,result2,cond3,result3,TRUE,result4)}$$

Both must be written with thresholds checked from highest to lowest (or lowest to highest, consistently), otherwise an early, looser condition catches values that should have been sorted into a later, stricter category first.

Worked example

Classify a gold grade (B2) into four tiers: High ($\ge2.0$ g/t), Medium ($\ge0.8$ g/t), Low ($\ge0.3$ g/t), Waste (below 0.3). Write the nested IF, and evaluate for B2 = 1.1.

$$\texttt{=IF(B2>=2.0,"High",IF(B2>=0.8,"Medium",IF(B2>=0.3,"Low","Waste")))}$$

$B2=1.1$: is it $\ge2.0$? No. Is it $\ge0.8$? Yes.

"Medium". Notice the check had to go from the highest threshold down, if the formula had checked B2>=0.3 first, a value of $1.1$ would have been caught early and wrongly labelled "Low," since $1.1$ also satisfies that looser condition, it's just never given the chance to be checked against the correct, higher tier first.

Practice 8

Using the worked example's formula, evaluate the classification for B2 = 2.5, B2 = 0.45, and B2 = 0.1.

Solution

2.5 → "High" ($\ge2.0$). $\ $ 0.45 → "Low" ($\ge0.3$ but $<0.8$). $\ $ 0.1 → "Waste" (below every threshold).

Practice 9

Rewrite the worked example's classification using IFS instead of nested IF.

Solution

=IFS(B2>=2.0,"High",B2>=0.8,"Medium",B2>=0.3,"Low",TRUE,"Waste")

The final TRUE,"Waste" pair acts as the catch-all default, exactly the role the innermost IF's false-branch played in the nested version, just written flatter and, most people find, easier to read once there are three or more tiers.

Practice 10

Classify rock hardness by unconfined compressive strength (UCS, MPa) in B3: Very Hard ($\ge200$), Hard ($\ge100$), Medium ($\ge50$), Soft (below 50). Write the nested IF formula.

Solution

=IF(B3>=200,"Very Hard",IF(B3>=100,"Hard",IF(B3>=50,"Medium","Soft")))

Practice 11

Evaluate Practice 10's formula for a UCS of 145 MPa.

Solution

$145\ge100$ but $145<200$: "Hard".

Practice 12

Someone writes the rock hardness classification checking thresholds from lowest to highest instead: =IF(B3>=50,"Medium",IF(B3>=100,"Hard",IF(B3>=200,"Very Hard","Soft"))). What does this formula return for a genuinely Very Hard sample, B3 = 250, and why is this wrong?

Solution

$250\ge50$ is checked first and is TRUE, so the formula immediately returns "Medium", before it ever gets the chance to check the higher, more correct thresholds. This is precisely the ordering trap the worked example warned about: once the loosest condition is satisfied, the formula stops and returns that result, it never continues checking the remaining, stricter conditions, regardless of how obviously wrong the classification is. Every sample with UCS $\ge50$ MPa, correct answer or not, gets funnelled into "Medium" by this broken formula.

Practice 13

In your own words, explain one genuine advantage of IFS over deeply nested IF statements, beyond just "it's shorter."

Solution

Nested IF statements accumulate closing parentheses that get genuinely hard to track once you're four or five levels deep, a single missing or misplaced parenthesis breaks the entire formula, and finding which one is wrong in a long nested chain is a real debugging headache. IFS lists condition/result pairs in a single flat sequence with no nesting at all, each pair reads almost like a sentence ("if this, then that"), which makes both writing it correctly the first time, and reading it back later to check the logic, noticeably easier once there are more than two or three tiers.

3 VLOOKUP

VLOOKUP searches down the first column of a reference table for a matching value, then returns a value from a specified column in that same row. This is the tool for cross-referencing two related tables, and the single most common real use in mining engineering is converting volume (BCM, bank cubic metres) to mass (tonnes) using a rock-type-specific density looked up from a reference table.

Key idea

$$\texttt{=VLOOKUP(lookup\_value, table\_array, col\_index\_num, range\_lookup)}$$

col_index_num counts columns from the left edge of the table array, starting at 1 for the lookup column itself. range_lookup: FALSE for an exact match (the usual choice for codes/IDs), TRUE or omitted for an approximate match (used for bracket/threshold lookups, Section 4 covers the difference in more detail).

Worked example

A reference table in E2:F5 lists rock type and density (t/BCM): Oxide 1.6, Transition 2.1, Fresh_Rock 2.8, Waste 2.4. Cell B2 contains Fresh_Rock and C2 contains a volume of 12,500 BCM. Write a formula in D2 for tonnes, using VLOOKUP to fetch the density, and evaluate it.

First, look up the density: =VLOOKUP(B2,$E$2:$F$5,2,FALSE), exact match on the rock type text, returning column 2 of the table (density).

Combine with the volume: $$\texttt{D2: =C2*VLOOKUP(B2,\$E\$2:\$F\$5,2,FALSE)}$$

VLOOKUP finds Fresh_Rock in column E, returns the matching density from column F: $2.8$ t/BCM.

$$12{,}500 \text{ BCM} \times 2.8 \text{ t/BCM} = 35{,}000 \text{ tonnes}.$$

Practice 14

Using the same density table (Oxide 1.6, Transition 2.1, Fresh_Rock 2.8, Waste 2.4), find the tonnes for 8,000 BCM of Oxide material.

Solution

$8000\times1.6 = 12{,}800$ tonnes.

Practice 15

In the density table (columns E=rock type, F=density), what is col_index_num for retrieving the density? What would col_index_num=1 return instead?

Solution

col_index_num=2 for density (the second column of the table array, counting from the lookup column itself as column 1). col_index_num=1 would just return the rock type text back again, the same value you searched with, since column 1 of the table array is always the lookup column itself.

Practice 16

An equipment spec table lists Model (column E) and Payload Tonnes (column F): CAT777 → 100, CAT785 → 150, CAT793 → 227. A truck log has model CAT793 in B2. Write a VLOOKUP formula in C2 to fetch its payload, and state the result.

Solution

=VLOOKUP(B2,$E$2:$F$4,2,FALSE) → 227 tonnes.

Practice 17

Explain the difference between range_lookup=FALSE and range_lookup=TRUE, using a royalty-rate table (grade bracket → royalty %) as an example of when the approximate (TRUE) version is actually the correct choice.

Solution

FALSE requires an exact match, correct for codes and names like rock type or truck model, where "close enough" makes no sense (there's no such thing as "approximately CAT793"). TRUE finds the largest value in the lookup column that is less than or equal to the search value, appropriate for bracket tables: a royalty table listing minimum grade thresholds (0, 0.5, 1.0, 2.0 g/t) with associated rates would use TRUE, so that a grade of 1.5 g/t correctly matches the "1.0" bracket (the largest threshold not exceeding 1.5) rather than failing to find an exact match on a number that will essentially never appear exactly in the lookup column. (Critical requirement for TRUE to work correctly: the lookup column must be sorted in ascending order.)

Practice 18

The rock-type density table lives in E2:F5. A VLOOKUP formula is written in D2 as =C2*VLOOKUP(B2,E2:F5,2,FALSE) (without dollar signs) and then filled down to D3, D4, D5. What goes wrong, and why?

Solution

Without the dollar signs ($E$2:$F$5), the table array reference is relative, and filling the formula down shifts it: D3's formula would reference E3:F6 instead of the original E2:F5, D4 would reference E4:F7, and so on, the lookup table itself appears to "slide down" the sheet along with the formula. Since rows 6 and 7 of the actual reference table are empty, most filled-down rows will return #N/A (no match found in a table that's now mostly blank), or worse, silently pick up wrong data if there happens to be something else sitting in those rows. Locking the table array with $ (absolute references) keeps it pinned to E2:F5 for every row the formula is filled into, exactly the fix.

Practice 19

A VLOOKUP formula returns #N/A for a rock type that you can visibly see listed correctly in the reference table. Give two realistic causes of this, beyond a genuinely missing entry.

Solution

(1) A trailing or leading space in one of the two text values, "Fresh_Rock " (with a space) does not exact-match "Fresh_Rock" as far as Excel is concerned, even though they look identical on screen. (2) Inconsistent capitalisation or underscore/space usage between the source data and the lookup table, e.g.\ "Fresh Rock" versus "Fresh_Rock", VLOOKUP with exact match requires the text to match character-for-character (though it is not case-sensitive), so a formatting inconsistency introduced during data entry is a very common, very frustrating real cause of an otherwise mysterious #N/A.

4 VLOOKUP pitfalls

VLOOKUP is powerful but has a genuine structural limitation worth knowing before you rely on it in a real workbook: it can only look for a match in the table's first column, and return a value from a column to its right. It cannot look backward, and cannot gracefully handle a missing match without extra help.

Key idea

VLOOKUP always searches the leftmost column of the table array and returns a value from a column with a higher index, i.e.\ further right. Wrap it in IFERROR to replace an ugly #N/A with a clean, deliberate fallback: $$\texttt{=IFERROR(VLOOKUP(...), "Not found")}$$

Worked example

An equipment table has Equipment_ID in column F and Model Name in column E (Model is to the left of ID). You need to look up a Model Name given an Equipment_ID. Can VLOOKUP do this directly? What are the options?

No, not directly. VLOOKUP always searches the first (leftmost) column of whatever range you give it as the table array, and here that would be column E (Model), not column F (ID), the exact opposite of what's needed.

Two real fixes: rearrange the table so ID sits to the left of Model (not always possible if the table is shared or generated elsewhere), or use INDEX/MATCH (a more flexible but more complex combination, beyond this lesson's scope), or, cleanest of all, switch to XLOOKUP.

This exact limitation is the single biggest reason XLOOKUP exists, covered next in Section 5.

Practice 20

A VLOOKUP formula returns #N/A for one specific truck ID out of 40, all other 39 work fine. You confirm the ID genuinely exists in the lookup table, correctly spelled, no extra spaces. What's a third possible cause, related to how the value was originally entered?

Solution

A data-type mismatch: the lookup value might be stored as text in one place and as a genuine number in the other (a classic case is an ID like 007, stored as the text string "007" in one column but auto-converted to the number 7 in the other, since Excel strips leading zeros from anything it interprets as numeric). VLOOKUP treats text "7" and number 7 as different values for exact-match purposes, even though they display identically or near-identically, producing a confusing single-row #N/A that looks, at a glance, exactly like every other correctly-matching row.

Practice 21

Rewrite =VLOOKUP(B2,$E$2:$F$5,2,FALSE) so that instead of displaying #N/A when a rock type isn't found, it displays the text "Check rock type".

Solution

=IFERROR(VLOOKUP(B2,$E$2:$F$5,2,FALSE),"Check rock type")

Practice 22

Explain why wrapping every VLOOKUP in IFERROR as a matter of habit is not always a good idea, even though it makes error messages disappear.

Solution

#N/A is genuinely useful information, it's telling you a real data problem exists (a typo, a missing entry, a formatting mismatch, exactly Practice 19 and 20's scenarios), and hiding it behind a friendly fallback text can mean that underlying problem gets quietly ignored rather than fixed. IFERROR is the right tool once you've deliberately decided a missing match is an expected, acceptable outcome (e.g.\ "this equipment simply hasn't been assigned a spec yet, and that's fine for now"), not a blanket habit to slap onto every lookup formula regardless of whether a missing match should actually be investigated.

Practice 23

Given the equipment table from the worked example (Model in E, ID in F, ID to the right of Model), write a VLOOKUP formula that correctly looks up a Model Name given an ID in B2, assuming the table columns cannot be rearranged.

Solution

There genuinely isn't a correct VLOOKUP formula for this, that's precisely the point of the worked example: VLOOKUP structurally cannot search a column and return a value to its left. Section 5's XLOOKUP is the direct, clean fix for exactly this situation, worth treating this problem as confirmation of the limitation rather than a puzzle to solve within VLOOKUP itself.

5 XLOOKUP: the modern replacement

XLOOKUP (available in current Excel and Excel for Microsoft 365, but not in older versions) fixes VLOOKUP's two biggest limitations directly: it can search and return in either direction, and it has a clean, built-in way to handle a missing match without needing a separate IFERROR wrapper.

Key idea

$$\texttt{=XLOOKUP(lookup\_value, lookup\_array, return\_array, if\_not\_found)}$$

No column-counting (col_index_num) required, you point directly at the column to search and the column to return, in any left/right relationship to each other. The optional fourth argument replaces the need for a separate IFERROR.

Worked example

Rewrite the density lookup (Section 3's worked example) as XLOOKUP, and separately, solve Practice 23's "look leftward" problem (Model Name given an ID, with ID sitting to the right of Model in columns E and F).

Density lookup: $$\texttt{=C2*XLOOKUP(B2,\$E\$2:\$E\$5,\$F\$2:\$F\$5)}$$ — functionally identical to the VLOOKUP version, just naming the search column (E) and return column (F) directly instead of counting.

Looking leftward (impossible for VLOOKUP): $$\texttt{=XLOOKUP(B2,\$F\$2:\$F\$4,\$E\$2:\$E\$4)}$$

Here the lookup array is column F (ID) and the return array is column E (Model), even though E sits to the left of F in the actual worksheet. XLOOKUP simply doesn't care which direction the return column is in, exactly the fix Practice 23 needed.

Practice 24

Rewrite the CAT793 payload lookup (Practice 16) using XLOOKUP instead of VLOOKUP.

Solution

=XLOOKUP(B2,$E$2:$E$4,$F$2:$F$4), returning the same 227 tonnes.

Practice 25

Rewrite Practice 21's IFERROR-wrapped VLOOKUP as a single clean XLOOKUP, using the built-in if_not_found argument.

Solution

=XLOOKUP(B2,$E$2:$E$5,$F$2:$F$5,"Check rock type")

One clean formula instead of a function wrapped inside another, the exact "not found" handling that took two nested functions in VLOOKUP is a single built-in argument here.

Practice 26

Given that XLOOKUP is strictly more capable than VLOOKUP, explain a genuine, practical reason a mining engineer would still need to know VLOOKUP well.

Solution

Not every workbook you'll touch was built recently, or in a current version of Excel, older company templates, files from clients or contractors on older software, and legacy spreadsheets that have been passed down and edited for a decade all commonly still use VLOOKUP, since XLOOKUP genuinely doesn't exist in older Excel versions and won't have been an option when those files were first built. Being able to read, debug, and extend an existing VLOOKUP-based workbook, not just build a new one, is a completely realistic day-one task, and you can't do that if you only ever learned XLOOKUP.

Practice 27 EXPLORE

For genuinely new work, where you know everyone opening the file has a modern Excel, would you default to VLOOKUP or XLOOKUP? Justify your answer in one or two sentences.

A reasonable answer

XLOOKUP, for new work with a modern-Excel audience: it's strictly more flexible (either-direction lookups), has cleaner error handling built in, and the formula itself is easier to read since you name columns directly rather than counting them. The only reason to reach for VLOOKUP by default would be compatibility with an older Excel version somewhere in the file's audience, worth actively checking before assuming XLOOKUP is safe to use.

6 Introduction to PivotTables

A PivotTable takes a large, flat table of raw records (one row per shift, per sample, per event) and summarises it instantly, total tonnes per pit, average grade per month, downtime hours per equipment type, by dragging field names into Rows, Columns, and Values boxes, no formulas typed at all.

Key idea

Drag a category field (like Pit, or Month) into Rows to group by it. Drag a numeric field (like Tonnes) into Values to summarise it, Excel defaults to Sum, but this can be changed to Average, Count, Min, Max, and more. A field in Columns creates a further breakdown running across the top of the table, a cross-tab.

Worked example

A raw daily production log has one row per shift: Pit, Month, Tonnes. Six rows exist: (North, Jan, 1200), (North, Jan, 1350), (South, Jan, 980), (North, Feb, 1400), (South, Feb, 1100), (South, Feb, 1050). Build (by hand, as a check on what the PivotTable would show) a summary of total tonnes by Pit and Month.

North, Jan: $1200+1350=2550$. $\ $ South, Jan: $980$.

North, Feb: $1400$. $\ $ South, Feb: $1100+1050=2150$.

JanFeb
North25501400
South9802150

This is exactly what a PivotTable with Pit in Rows, Month in Columns, and Sum of Tonnes in Values produces automatically, no formula written anywhere, and it updates instantly if the raw data changes and the PivotTable is refreshed.

Practice 28

Using the worked example's table, what is the total across both pits for January?

Solution

$2550+980 = 3530$ tonnes, the "Jan" column total a PivotTable would automatically show as a grand total row/column.

Practice 29

You want a PivotTable showing total tonnes per pit only, with no month breakdown at all. Which field(s) go in Rows, and which in Values?

Solution

Pit in Rows, Tonnes in Values (Sum). Month is simply left out of the PivotTable entirely, since it's not needed for this particular summary, Excel automatically aggregates across every month for each pit when Month isn't placed anywhere in the layout.

Practice 30

The Values field is changed from Sum to Average. Using the worked example's raw data, what would the North/Jan cell show instead of 2550?

Solution

North had two shifts in January (1200 and 1350): average $=\dfrac{1200+1350}2=1275$ tonnes per shift, a genuinely different, and often more useful, question than the raw total ("how big was a typical shift" versus "how much did we produce in total").

Practice 31

Two new rows of raw data are added to the source table after the PivotTable has already been built. Does the PivotTable update automatically? What action, if any, is required?

Solution

No, not automatically, a PivotTable is a snapshot summary computed at the time it was built (or last refreshed), it does not watch the source data live. You need to explicitly click Refresh (right-click the PivotTable, or use the Refresh button on the ribbon) for it to re-scan the source range and incorporate the new rows. This is exactly why building the source data as a genuine Excel Table (from the previous lesson) is worth the habit, a Table-based source range that grows automatically means the PivotTable's data range keeps up without needing to be manually re-selected every time rows are added, only Refresh is still required.

Practice 32

Explain the genuine advantage of a PivotTable over writing five separate SUMIF-style formulas to compute the same five totals (one per pit-month combination) manually.

Solution

Beyond simply being faster to set up (drag two fields versus writing and debugging several conditional-sum formulas), a PivotTable is trivially re-shaped: switching from "Pit by Month" to "Month by Pit," or adding a third breakdown by Equipment, or changing Sum to Average, is a few clicks of dragging fields around, not a rewrite of every formula. It also automatically produces grand totals and subtotals correctly with no risk of a formula referencing the wrong range, and it scales effortlessly to genuinely large datasets (thousands of raw rows) where hand-writing individual summary formulas for every category combination would become completely impractical.

Practice 33

Propose a PivotTable layout (which fields in Rows, Columns, Values, and which summary statistic) for a raw equipment downtime log with columns Equipment_Type, Reason_Code, and Downtime_Hours, to answer: "which reason code causes the most total downtime for each equipment type?"

Solution

Equipment_Type in Rows, Reason_Code in Columns, Sum of Downtime_Hours in Values, producing a cross-tab where each row (equipment type) can be scanned across its columns (reason codes) to immediately spot the largest number, the dominant downtime cause for that specific equipment type, exactly the kind of question a PivotTable answers in seconds that would otherwise need careful manual filtering and summing.

7 Explore: putting it all together

These three tools are almost never used in isolation in a real workbook, IF classifies a row, VLOOKUP/XLOOKUP enriches it with data from another table, and a PivotTable summarises the result. The following are open-ended builds, worth actually opening Excel for.

Practice 34 EXPLORE

Build a small block model: 10 rows of made-up sample data (Sample_ID, Rock_Type, Grade_gt, Volume_BCM). Add a Density lookup column (XLOOKUP against a 4-row rock-type/density table), a Tonnes column, and an Ore/Waste column (IF against a 0.5 g/t cutoff). Then build a PivotTable summarising total tonnes by Rock_Type and by Ore/Waste classification.

What this ties together

This single small workbook uses all three tools from this lesson in the exact sequence they're actually used in real resource estimation: XLOOKUP enriches each raw sample row with a density (Section 3/5), a formula converts volume to tonnes using that looked-up density, IF classifies each row as ore or waste (Section 1), and the PivotTable (Section 6) then summarises the whole block model into a single, decision-ready number: how many tonnes of ore versus waste, broken down by rock type. This is, genuinely, a simplified version of what real mine planning software does under the hood.

Practice 35 EXPLORE

Build a royalty calculator: a grade-bracket table (4 brackets, ascending, with royalty rates), then a sample list where each row's royalty rate is found via an approximate-match VLOOKUP or XLOOKUP, and a royalty-owed column (grade × rate × some fixed tonnage). Total it with SUM.

What to watch for

The bracket table must be sorted ascending by the minimum grade threshold for approximate-match lookup to work correctly (Practice 17's requirement), a genuinely easy thing to get wrong if the table is later re-sorted for display purposes without remembering why the order mattered in the first place. Worth deliberately testing a grade value sitting exactly on a bracket boundary, to confirm which bracket it lands in, exactly the boundary-condition discipline built in Practice 1 and Practice 3.

Practice 36 EXPLORE

Build an equipment cost-per-hour report: an equipment spec table (Model, Payload, Fuel_Burn_Rate, Hourly_Cost), a fleet log (Truck_ID, Model, Hours_Run) with Hourly_Cost pulled in via lookup and a Total_Cost column computed, then a PivotTable summarising Total_Cost by Model.

What this represents

This is a realistic, simplified fleet cost report: one reference table (specs, rarely changing) joined against an operational log (hours run, changing daily) via lookup, exactly the "master table + transaction table" pattern that underlies most real production and cost systems, whether built in Excel, a database, or eventually pandas. Recognising this pattern, a small reference table joined to a larger operational log, is one of the most transferable ideas in this entire lesson.

Practice 37 EXPLORE

Build a QAQC batch tracker: 15 rows of original/duplicate assay pairs, a percentage-difference column, a Pass/Fail IF flag (10% threshold), and a PivotTable counting how many samples passed versus failed per lab batch (add a Batch_ID column, 2–3 batches).

What this represents

Counting Pass/Fail per batch in a PivotTable uses the Count value-field setting rather than Sum or Average (Section 6's key idea mentioned this option without dwelling on it, this is the natural place to actually use it): dragging the Pass/Fail field itself into Values and setting it to Count gives, per batch, how many samples fell into each category. A batch with an unusually high fail rate is exactly the kind of signal a QAQC officer would want flagged immediately, and this small workbook produces that signal automatically from raw duplicate-pair data.

What this is really testing

There's no single expected answer, but the pattern worth noticing is consistent across all four builds: raw, granular field data (one row per sample, per truck-hour, per assay pair) gets enriched with reference information via lookup (density, cost rate, royalty rate), then classified with IF logic (ore/waste, pass/fail), and finally aggregated into a PivotTable that answers the actual business question a manager cares about (total ore tonnes, total fleet cost, batch pass rate). The bridge to data science

The IF column becomes roughly df['Ore_Waste'] = np.where(df['Grade_gt']>=0.5,'Ore','Waste'), the exact same true/false branching logic, just as a vectorised operation applied to an entire column at once rather than one cell at a time. The PivotTable becomes df.groupby(['Rock_Type','Ore_Waste'])['Tonnes'].sum(), Rows and Values from the PivotTable interface become arguments to groupby() directly. Neither of these lines does anything conceptually new compared to what you just built by hand in Excel, they're the same three ideas (classify, enrich, aggregate) with different syntax, exactly the payoff this lesson series has been building toward from the very first "why Excel for an engineer" section.
