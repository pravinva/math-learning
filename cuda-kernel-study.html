Memory model and manual memory managementMaster this
Stack vs heap, new/delete, RAII, smart pointers, alignment
Week 1
10 hrs
▼

CUDA kernels live and die by memory. Before you write a single GPU line, you need to own C++ memory at the lowest level — because on the GPU, you are the allocator and there is no GC.

  • Understand stack vs heap allocation. Stack is fast, size-limited, auto-freed. Heap is flexible, manual.
  • Practice raw new / delete and new[] / delete[] — CUDA memory APIs use raw pointers everywhere
  • RAII: destructors, copy constructors, move constructors — prevent leaks without GC
  • Memory alignment: alignas, struct padding, cache line size (64 bytes on x86). GPU memory has similar constraints.
  • Smart pointers: unique_ptr, shared_ptr — understand what they do, and why you often cannot use them at GPU boundaries

GPU kernels work with flat memory. Matrices are stored as 1D arrays. Pointer arithmetic needs to be instinctive.

  • Pointer arithmetic: ptr + n advances by n * sizeof(T) — drill until automatic
  • 2D matrix as 1D: element [i][j] = index i * cols + j. This is the most-used formula in kernel code.
  • Const correctness: const T* vs T* const vs const T* const — critical for CUDA kernel parameters
  • void* and casting — appears constantly in CUDA memory APIs (cudaMalloc(void**))
  • Function pointers — appears in CUDA callback and dispatch patterns

How a GPU differs from a CPUMust understand
SIMT, SMs, warps, latency hiding vs latency avoidance
Week 1
8 hrs
▼

A CPU minimises latency for a single thread. A GPU maximises throughput across thousands of threads. This is the core mental shift everything else builds on.

  • Streaming Multiprocessors (SMs): the GPU equivalent of a CPU core. RTX 3090 has 82 SMs. An A100 has 108. Each SM runs many threads concurrently.
  • Warps: groups of 32 threads that execute in lockstep (SIMT). If threads diverge at a branch, both paths execute serially — performance drops.
  • Latency hiding: GPUs issue memory requests and switch to other warps while waiting. Requires enough active warps to keep the SM busy.
  • Key numbers to memorise: warp size = 32, RTX 3090 peak bandwidth = 936 GB/s, A100 = 2 TB/s HBM2e

The CUDA programming model — threads, blocks, gridsWrite yourself
Kernel launch syntax, thread indexing, nvcc, your first kernel
Week 1–2
10 hrs
▼

CUDA extends C++ with a small set of keywords. Learn the syntax, then understand what it maps to in hardware.

  • Thread hierarchy: thread → block → grid. Threads in a block share memory and can synchronise. Blocks cannot directly communicate.
  • Launch config: kernel<<<gridDim, blockDim>>>(args)
  • Thread indexing: int i = blockIdx.x * blockDim.x + threadIdx.x — memorise this formula
  • nvcc: CUDA compiler. Splits host (CPU) and device (GPU) code. Compiles device code to PTX then SASS.

3
The GPU memory hierarchyConcepts first
Global, shared, constant, registers — bandwidth, size, scope
Week 2–3
7 hrs
▼

Memory is where GPU performance is won or lost. The different memory types have drastically different speeds, sizes, and scopes.

  • Global memory: largest, slowest (936 GB/s on RTX 3090), accessible by all threads. Your main working memory.
  • Shared memory: fast (~19 TB/s equivalent on-chip), small (48–100KB per SM on RTX 3090), shared within a block. The key to fast kernels — manually managed L1 cache.
  • Registers: fastest, private per thread, limited (~255 per thread). Spilling to local memory (slow) happens when exceeded.
  • Constant memory: 64KB read-only cached — for kernel parameters that all threads read identically.

Memory coalescing and access patternsMust profile yourself
The single most impactful global memory optimisation

▼

When threads in a warp access consecutive memory addresses, the GPU services them in a single transaction. When they do not, performance degrades up to 32x.

  • 32 threads accessing A[threadIdx.x] through A[threadIdx.x + 31]: coalesced — one transaction
  • Same 32 threads accessing every 32nd element: uncoalesced — 32 separate transactions

  • Understand Nsight Compute metric: l1tex__t_sectors_pipe_lsu_mem_global_op_ld_lookup_hit — the coalescing efficiency score

Shared memory and bank conflictsHybrid
32 banks, bank conflicts, padding fix, broadcast exception
Week 2
8 hrs
▼

Shared memory is divided into 32 banks. Multiple threads in a warp accessing the same bank simultaneously causes serialisation — a bank conflict.

  • Thread i maps to bank i % 32
  • Common conflict: accessing column of a 2D shared array (strided access). Fix: pad width by 1.
  • Broadcast exception: all threads reading the same address in the same bank — no conflict, broadcasts to all.

Roofline model and Nsight Compute profilingInterpret yourself
Arithmetic intensity, compute-bound vs memory-bound, Nsight workflow
Week 3–4
10 hrs
▼

The roofline model tells you whether your kernel is limited by compute or by memory bandwidth. Everything you optimise should be driven by this answer first.

  • Arithmetic intensity: FLOPs / bytes accessed. High = compute-bound. Low = memory-bound.
  • A 1024x1024 GEMM: ~2B FLOPs, ~12MB memory = 167 FLOP/byte. Compute-bound on any modern GPU.
  • ReLU: 1 FLOP per element, 8 bytes read+write = 0.125 FLOP/byte. Heavily memory-bound.
  • Nsight Compute generates a roofline chart automatically — use ncu --set roofline
